# Phase 3: Data Dictionary, Integration Testing, Deployment Runbook - Specification

## Overview

Implement operational and testing documentation to enable database understanding, cross-service testing, and production deployment procedures. Three interconnected deliverables focused on operations and quality assurance.

**Goals:**
- Provide complete database schema documentation for developers and DBAs
- Enable comprehensive testing across microservice boundaries
- Create clear, step-by-step deployment and recovery procedures
- Reduce operational friction and errors in production

---

## Component 1: Data Dictionary

### Deliverable
- **File**: `docs/database/01-data-dictionary.md`
- **Scope**: 1,000-1,200 lines
- **Target Audience**: Developers, DBAs, Data Analysts

### Content Structure

1. **Overview & Purpose**
   - What is the data dictionary
   - How to use it
   - Data model at a glance
   
2. **Core Entities** (for each entity)
   - Entity name and purpose
   - Relationships to other entities
   - Columns with types, constraints, descriptions
   - Indexes and performance notes
   - Example queries
   
3. **Entities by Service**
   - **Auth Service**: users, roles, permissions, sessions, tokens
   - **Training Service**: exercises, disciplines, routines, workouts, sessions
   - **Tracking Service**: diet_logs, weight_logs, metrics, goals, progress
   - **Notification Service**: notifications, preferences, delivery_logs, templates

4. **Relationships Diagram**
   - Entity relationship diagram (ASCII or reference)
   - Foreign key relationships
   - Cardinality notes
   
5. **Queries Reference**
   - Common queries (get user, list exercises, etc.)
   - Complex queries (analytics, reports)
   - Performance tips
   
6. **Data Integrity Rules**
   - Constraints and validations
   - Cascading rules
   - Data quality standards

### Specific Entities to Document

**Auth Service:**
- `users` - User accounts with email, password_hash, created_at, updated_at
- `roles` - admin, trainer, user
- `permissions` - read, write, delete by resource
- `user_roles` - Junction table
- `sessions` - Login sessions with JWT tokens

**Training Service:**
- `disciplines` - chest, back, legs, etc.
- `exercises` - Individual exercises with descriptions
- `routines` - Pre-built workout routines
- `workouts` - User workout sessions
- `workout_exercises` - Exercises in a workout session

**Tracking Service:**
- `diet_logs` - Daily food/calorie tracking
- `weight_logs` - Daily weight measurements
- `metrics` - User statistics (avg calories, weight trend)
- `goals` - User goals (weight, calories, etc.)
- `progress` - Goal progress tracking

**Notification Service:**
- `notifications` - Notification records
- `notification_preferences` - User notification settings
- `delivery_logs` - Delivery status tracking
- `notification_templates` - Email/SMS templates

## Component 3: Deployment Runbook

### Deliverable
- **File**: `docs/deployment/01-deployment-runbook.md`
- **Scope**: 600-800 lines
- **Target Audience**: DevOps, Operators, Release Managers

### Content Structure

1. **Pre-Deployment Checklist**
   - Code review completed
   - Tests passing (unit + integration)
   - Database migrations reviewed
   - Secrets updated if needed
   - Release notes prepared
   
2. **Deployment Steps**
   - Verify staging environment
   - Build Docker images
   - Tag and push to registry
   - Update Kubernetes manifests
   - Apply deployments
   - Monitor health checks
   
3. **Deployment Strategies**
   - Rolling deployment (default, zero-downtime)
   - Canary deployment (test with small % of traffic)
   - Blue-green deployment (full cutover)
   
4. **Database Migrations**
   - Run migrations before deployment
   - Verify migration success
   - Rollback procedures
   - Data validation post-migration
   
5. **Monitoring During Deployment**
   - Kubernetes pod status
   - Service health checks
   - Application logs
   - Error rates and latency
   - Database connections
   
6. **Verification Steps**
   - Health endpoints responding
   - API endpoints working
   - Database queries successful
   - No error rate spikes
   - Performance within baseline
   
7. **Rollback Procedures**
   - When to rollback
   - Kubernetes rollback command
   - Database rollback (if needed)
   - Communication to team
   
8. **Common Issues & Solutions**
   - Pod crash loop
   - Deployment timeout
   - Database migration failure
   - Insufficient resources
   
9. **Post-Deployment**
   - Smoke tests
   - Monitor for 30 minutes
   - Update status page
   - Create deployment record
   
10. **Recovery Procedures**
    - Data loss recovery (from backups)
    - Service restoration
    - Communication timeline

### Specific Procedures

1. **Emergency Rollback**
   - Detect critical issue
   - Initiate rollback (< 5 minutes)
   - Verify rollback success
   - Root cause analysis

2. **Database Emergency**
   - Identify data corruption
   - Restore from backup
   - Validate data integrity
   - Redeploy application

3. **Multi-Region Failover**
   - Detect primary region failure
   - Activate standby region
   - Update DNS/routing
   - Verify all services running

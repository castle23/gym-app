### Part B: Create Changelog Procedures

**File**: `docs/deployment/02-changelog-procedures.md`

**Content Plan** (300-400 lines):

### 1. Understanding Semantic Versioning (80 lines)

```
MAJOR.MINOR.PATCH (e.g., 1.2.3)

MAJOR - Breaking changes (users must update code)
  - Auth token format changes
  - API endpoint removal
  - Database schema changes requiring migration

MINOR - New features (backward compatible)
  - New API endpoints
  - New optional fields
  - Database schema additions (new tables)

PATCH - Bug fixes (backward compatible)
  - Performance improvements
  - Security patches
  - Bug fixes with no API changes

Pre-release versions: v1.0.0-alpha, v1.0.0-beta, v1.0.0-rc.1
```

### 2. Writing Changelog Entries (100 lines)

```
Categories:
- Added: New features, new services
- Changed: Behavioral changes, improvements
- Deprecated: Features to be removed in future
- Removed: Features that were deprecated
- Fixed: Bug fixes
- Security: Security improvements and patches

Guidelines:
- Write from user perspective (not developer)
- Be specific about what changed
- Link to related GitHub issues/PRs
- Highlight breaking changes clearly

Examples:

✅ Good: "Added GET /api/training/workouts/{id} endpoint for fetching individual workout details"
❌ Bad: "Added new endpoint"

✅ Good: "Fixed user authentication timeout when token refresh rate exceeded #234"
❌ Bad: "Fixed token bug"

✅ Good: "⚠️ BREAKING: Changed authentication header format from 'Authorization: Bearer X' to 'X-Auth-Token: X'"
```

### 3. Maintaining Unreleased Section (80 lines)

```
Process during development:
1. Create PR with changes
2. Add changelog entry to [Unreleased] section
3. Place in appropriate category (Added, Fixed, etc.)
4. Link to PR number
5. Request review

Example PR entry:
```
### Added
- New endpoint: POST /api/training/routines for creating custom workout routines (#456)
- Support for multiple exercise variants per exercise (#456)
```

Preventing duplicates:
- One entry per logical change, not per commit
- If multiple PRs contribute to same feature, combine in changelog
- Use filtering to find existing entries
```

### 4. Release Process (80 lines)

```
Week of Release:
1. Code freeze (no new features)
2. Create release branch: git checkout -b release/v1.1.0
3. Update version numbers in code (pom.xml, package.json, docker tags)
4. Finalize changelog entries (move from Unreleased to vX.Y.Z)
5. Create git tag: git tag -a v1.1.0 -m "Release v1.1.0"
6. Push to GitHub: git push origin release/v1.1.0 && git push origin v1.1.0
7. Create release notes on GitHub matching changelog
8. Run smoke tests in production
9. Monitor for issues

Emergency release:
- Same process but prioritized and communicated
- Example: v1.0.1 for critical security patch
```

### 5. Breaking Changes & Migration (60 lines)

```
For MAJOR version releases:

Include migration guide:
- What changed
- Why it changed
- Step-by-step migration instructions
- Deprecation timeline (usually 2-3 releases)

Example breaking change entry:
```
### Changed
- **BREAKING**: JWT token format changed from HS256 to RS256
  - Migration: See MIGRATION_v2.0.0.md for step-by-step instructions
  - Timeline: v1.9.0 deprecated HS256, v2.0.0 removes support
  - No action needed if using token refresh endpoint
